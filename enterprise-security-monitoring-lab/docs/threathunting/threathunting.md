
# Threat Hunting

## Overview

Threat hunting is the proactive process of searching through security logs and telemetry to identify malicious activity that may not trigger automated alerts. Unlike detection rules, which generate alerts when predefined conditions are met, threat hunts begin with a hypothesis and use log analysis to validate or disprove that hypothesis.

For this project, threat hunting was performed using Apache web server logs ingested into Splunk Enterprise. The objective was to identify suspicious web activity targeting the DVWA application and reconstruct attacker behavior.

---

# Hunt 1 – Attacker Activity

## Objective

Identify all HTTP requests generated by a specific attacker IP address.

## Hypothesis

A single source IP may perform multiple actions against the web application during an attack.

## SPL

```spl
index=main clientip=192.168.52.10
| table _time clientip method uri status useragent
| sort _time
```

## Findings

The hunt returned over 200 HTTP requests originating from the Kali Linux attacker system (192.168.52.10). The activity included authentication attempts, navigation through DVWA pages, exploitation attempts, and file upload activity.

## Analyst Notes

This hunt provides a complete chronological view of an individual attacker's behavior and serves as the starting point for incident investigations.

# Hunt 2 – Multi-Stage Attack

## Objective

Determine whether a single attacker executed multiple attack techniques during one attack session.

## Hypothesis

An attacker often performs several attack techniques rather than a single isolated event.

## SPL

```spl
index=main clientip=192.168.52.10
| eval attack_type=case(
    like(uri,"%brute%"),"Brute Force",
    like(uri,"%sqli%"),"SQL Injection",
    like(uri,"%exec%"),"Command Injection",
    like(uri,"%upload%"),"File Upload",
    true(),"Other"
)
| search attack_type!="Other"
| table _time clientip attack_type method uri status
| sort _time
```

## Findings

The investigation confirmed multiple attack techniques originating from the same source IP:

- Brute Force
- Command Injection
- File Upload
- SQL Injection

![Alt text](../screenshots/Screenshot%202026-07-07%20181017.png)

## Analyst Notes

Correlating multiple attack techniques from a single source provides stronger evidence of malicious activity than evaluating each event independently.

# Hunt 3 – User Agent Analysis

## Objective

Identify the software or browsers generating HTTP requests.

## Hypothesis

Attack tools often use distinctive User-Agent strings that differ from normal browser traffic.

## SPL

```spl
index=main
| stats count by useragent
| sort -count
```

## Findings

User-Agent analysis identified the browsers and tools used during the lab. While this environment primarily generated browser-based traffic, this hunt can also detect automated scanners and exploitation tools.

![Alt text](../screenshots/Screenshot%202026-07-07%20182028.png)

## Analyst Notes

User-Agent analysis helps distinguish legitimate user activity from automated attack tools.

# Hunt 9 – Attack Type Distribution

## Objective

Summarize attack activity by technique.

## Hypothesis

Different attack types can be categorized to understand the overall attack profile.

## SPL

```spl
index=main clientip=192.168.52.10
| eval attack_type=case(
    like(uri,"%brute%"),"Brute Force",
    like(uri,"%sqli%"),"SQL Injection",
    like(uri,"%exec%"),"Command Injection",
    like(uri,"%upload%"),"File Upload",
    like(uri,"%login%"),"Login Activity",
    true(),"Other"
)
| search attack_type!="Other"
| stats count by attack_type
| sort -count
```

## Findings

Attack activity was grouped into logical categories, making it easier to understand which techniques were used most frequently.

![Alt text](../screenshots/Screenshot%202026-07-07%20184134.png)

## Analyst Notes

Categorizing attacks improves reporting, dashboard visualization, and trend analysis.

# Hunt 10 – Abnormal Status Codes

## Objective

Identify HTTP responses indicating errors or abnormal application behavior.

## Hypothesis

HTTP status codes outside the successful (2xx) range may indicate failed requests, probing, or exploitation.

## SPL

```spl
index=main status>=400
| stats count by status uri clientip
| sort -count
```

## Findings

The hunt identified abnormal HTTP responses for further investigation.

![Alt text](../screenshots/Screenshot%202026-07-07%20184330.png)

## Analyst Notes

Reviewing abnormal status codes helps identify unsuccessful attacks, application issues, and potential reconnaissance activity.

---

# Summary

This threat hunting phase demonstrated how Apache web server logs can be used to proactively identify suspicious behavior beyond traditional alerting. By combining multiple hunts, it was possible to reconstruct attacker activity, categorize attack techniques, identify targeted resources, analyze request patterns, and visualize attack timelines.

These hunts complement the detection rules created earlier and provide a repeatable methodology for investigating future web application attacks.
